核心矛盾在于：**我们无法在一个分布式系统中简单地实现一个原子操作，来同时更新位于两个不同节点（MySQL 和 Redis）的数据。**

# cache aside

最终一致性

这是最常用、最基础的策略。应用程序直接与缓存和数据库交互。

- 读流程：

1. 接收读请求。
2. 先查 Redis，如果存在数据（Hit），直接返回。
3. 如果 Redis 中没有数据（Miss），则从 MySQL 中查询。
4. 将 MySQL 中查询到的数据写入 Redis，然后返回数据。

- 写流程：

1. 接收写请求。
2. 更新 MySQL 中的数据。
3. 删除 Redis 中对应的缓存。

为什么是删除（Delete/Invalidate）而不是更新（Update）缓存？

- 性能：如果缓存数据是经过复杂计算得出的，直接删除比更新更轻量。特别是写多读少的场景，频繁更新缓存可能白费功夫（数据可能很快又被修改或根本没人读）。
- 一致性：在并发写场景下，更新缓存的顺序可能和更新数据库的顺序不一致，导致缓存脏数据。而删除操作是幂等的，更安全。

优点：

- 简单直观，易于理解和实现。
- 缓存命中率由业务逻辑决定，比较可控。

缺点：

- 首次请求一定缓存未命中，有延迟。
- 存在短暂的不一致窗口：在更新 MySQL 后、删除 Redis 前，或者删除 Redis 后、新缓存写入前，读请求可能会读到旧数据。
- 需要处理缓存穿透、击穿、雪崩等问题。

## 1. 延迟双删（Delayed Double Delete）

由于 Cache Aside 最常用，我们重点讨论如何优化它的一致性。

用于解决在 Cache Aside 下，高并发场景中因操作时序问题导致的长时间脏数据问题。

场景：

1. 线程 A 更新 MySQL。
2. 线程 B 读取数据，发现缓存失效，从 MySQL 读到A更新前的旧数据，并准备写入 Redis。
3. 线程 A 删除 Redis。
4. 线程 B 将旧数据写入 Redis。 -> 导致 Redis 中一直是脏数据。

解决方案（延迟双删）：

1. 更新 MySQL 数据。
2. 第一次删除 Redis 缓存。
3. 等待一个短暂的时间（如几百毫秒）。这个时间 = 读数据库耗时 + 主从同步耗时 + 一些缓冲时间。
4. 第二次删除 Redis 缓存。

第二次删除的目的是为了清除在那段“不一致窗口”内可能由其他并发读请求写入的脏数据。

## 2. 通过 MySQL Binlog 异步淘汰/更新缓存

这是业界最成熟、最可靠的方案之一。它将缓存与数据库的一致性同步，从业务代码中解耦出来。

核心思想：利用 MySQL 的 Binlog（二进制日志，用于主从复制和数据恢复）作为“数据变更”的流。通过一个中间件（如 Canal、Debezium）来订阅 Binlog，解析出哪些数据发生了变更，然后异步地删除或更新 Redis 中对应的缓存。

工作流程：

1. 业务代码正常更新 MySQL。
2. MySQL 将更新操作记录到 Binlog。
3. 缓存同步中间件（如 Canal）模拟 MySQL 从库，拉取并解析 Binlog。
4. 中间件获取到变更的数据和 Key，向 Redis 发送删除命令。

优点：

- 业务代码极度简化，写操作不再关心缓存。
- 数据同步延迟低，通常在毫秒级别。
- 高可靠性，基于 MySQL 主从复制机制，非常稳定。

缺点：

- 引入了新的中间件，系统架构变复杂，运维成本增加。
- 仍然是最终一致性。

# read/write through

强一致性。

应用程序将缓存视为主要数据源，缓存组件自己负责与数据库交互。

- 读流程：

1. 应用程序查询缓存。
2. 缓存命中则返回。
3. 缓存未命中，由缓存组件自己从 MySQL 加载数据，填入缓存后返回。

- 写流程：

1. 应用程序更新缓存。
2. 由缓存组件负责同步地更新 MySQL 和 Redis。

优点：

- 应用程序逻辑更简单，将一致性责任委托给缓存组件。

缺点：

- 实现复杂，通常需要支持该模式的缓存中间件（如一些商业版的分布式缓存）。
- 写操作必须等待缓存和数据库都更新完成，延迟较高。

# write behind

这是 Write Through 的异步版本。

- 写流程：

1. 应用程序更新缓存。
2. 缓存立即返回成功。
3. 缓存组件异步地、批量地将数据更新到 MySQL。

优点：

- 写性能极高，延迟极低。
- 可以对数据库写操作进行合并，减轻数据库压力。

缺点：

- 有数据丢失风险：如果缓存宕机，尚未刷盘的数据会丢失。
- 一致性最弱，数据库和缓存在大部分时间都是不一致的。
- 实现最为复杂。
