# 为什么要数据建模？

1. 质量，中间层收口，落地 single source of truth，统一数据团队专业维护，保持数据质量
2. 成本与效率，数据复用，降低成本，提高效率
3. 稳定性与性能保障

# 范式建模

在数据库场景使用，目的是简洁、高性能

- 模型：ER模型，实体表、关系表
- 建模原则：3NF(三范式)

## ER模型-三范式

1. 属性必须是原子性的，不可拆分。
2. 属性必须全部依赖主键，不能部分依赖。比如订单表：订单ID、用户ID、商品ID，用户手机号、商品价格不能作为属性
3. 属性必须只依赖主键，不能依赖其他属性。比如用户表：用户ID，城市，邮编则不能作为属性

# 维度建模

在数仓场景使用

- 模型：RD模型，事实表、维度表，事实表：事件事实表、状态事实表
- 建模原则：建议星型模型(三退化)，雪花模型辅助(维度表继续关联其他维度表)

## 星型模型-三退化

1. 多个维表退化到一个维表中，避免关联
2. 维表直接退化到事实表，避免关联
3. 多个事实表退化到一个事实表中，宽表提高使用效率、降低复杂度

- 要考虑使用场景，共同使用场景属性才需要做成宽表，相对个性属性不宜进入宽表
- 要考虑使用频率，非高频场景属性不宜进入宽表，导致过度冗余
- 要考虑产出时效性，不要差别过大，导致整体拖慢

## 建模流程

1. 业务过程梳理，维度表和事实表识别和划分。
2. 事实表和维度表属性详细设计
3. 模型Review
4. 开发

### 维度表建模

几个关键问题

1. 维度表唯一
2. 勇于退化

- 水平拆分，当次要属性过多，保持核心属性，其他属性做成独立维度表
- 垂直拆分，按照使用场景、频率和时效性，划分主属性和拓展属性
